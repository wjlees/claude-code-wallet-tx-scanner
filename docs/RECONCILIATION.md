# 지갑 잔고대사(Reconciliation) 시스템 개요

> 목적: 함께 개발하는 사람이 "이미 있는 것 / 만들 것 / 왜"를 한 번에 잡을 수 있는 전체 그림.
> 설계가 확정되는 대로 이 문서에 구체화한다. (monorepo 반영 규칙은 [MONOREPO-MAPPING.md](./MONOREPO-MAPPING.md))

## 배경

거래소 내부 원장(OMS)이 말하는 잔고와 실제 블록체인 위의 지갑 잔고는 항상 일치해야 한다.
이 시스템은 그 일치를 **자동으로 검증**하고, 불일치가 생기면 **원인 tx까지 추적**할 수 있게 하는 것이 목표다.

전제가 되는 수집기는 이미 있다: **wallet-tx-scanner** 가 감시 지갑의 모든 in/out 트랜잭션을
`detected_transactions`(wallet DB)에 쌓고 있다(= 아래의 "온체인TX"). 이번에 만드는 것은 그
데이터를 **활용하는 쪽**(스냅샷·대사·리포트·웹)이다.

## 용어

| 용어 | 정의 |
|---|---|
| **DB잔고** | OMS 스냅샷에서 온 잔고. 내부 원장 기준 "지갑에 있어야 할 양" (transaction_status 별 구분이 있을 수 있음 — 확정분/처리중 구분 확인 필요) |
| **온체인잔고** | 노드 `getBalance` 로 실측한 지갑 잔고. UTXO(BTC/BCH)는 getBalance 가 없으므로 `wallet_scanner_unspents` 테이블의 SUM(usable=1) 사용 |
| **온체인TX** | 핫월렛/콜드월렛/회사지갑(+UTXO 는 유저 입금주소 포함)의 in/out 트랜잭션 전부. wallet-tx-scanner 가 `detected_transactions` 에 적재 중 |

## 만들 것 (5개)

### 1. 온체인잔고 스냅샷 찍기
(현재 data 팀이 5분마다 찍는 것을 우리가 직접)

- 주기는 5분일 필요 없음 — **1시간마다 실측**하고, 중간 시점이 필요하면 온체인TX 로 계산하면 됨.
- a. 잔고를 저장할 때 **그 잔고가 유효한 시점의 block_number 를 반드시 함께 저장**한다.
  이게 없으면 시점 보정(2번)이 불가능하다.
- b. block_number 가 있으면 "스냅샷 ± 온체인TX" 로 **임의 블록 시점의 잔고를 계산**할 수 있다.
- c. **헬스체크(자기 검증)**: `이번 실측 잔고` vs `직전 실측 잔고 + 그 사이 온체인TX 합산` —
  이 둘은 당연히 같아야 한다. 다르면 둘 중 하나가 고장난 것(스캐너가 tx 를 놓쳤거나, 잔고
  실측이 잘못됐거나). **이 등식이 깨지지 않는 한 "온체인TX 데이터는 완전하다"고 믿을 수 있고,
  그래야 아래 대사 결과도 믿을 수 있다** — 단순 헬스체크가 아니라 대사 전체의 전제 조건.

### 2. DB잔고와 온체인잔고의 시점 맞추기

- DB잔고 스냅샷 시각과 온체인잔고 실측 시각은 정확히 일치하지 않는다(정각에서 몇 초~몇 분 어긋남).
- a. 온체인TX 로 그 어긋난 구간의 입출금을 **보정 가산/차감**해서 두 잔고를 같은 시점으로
  환산한 뒤 비교한다. (1-a 의 block_number 가 이 보정의 기준점)

### 3. 불일치 감지 → 알림/리포트

- 시점을 맞췄는데도 잔고가 다르면, 온체인TX 중 **OMS 에 대응 기록이 없는 tx_id**(또는 그 반대)를
  찾아 원인으로 지목한다.
- a. 단, 재경팀이 수행하는 **정상적인 외부 이동**(예: 바이낸스 지갑 → 핫월렛)은 OMS 에 없어도
  정상이다 — 재경팀이 tx_id 를 등록해두면 대사에서 제외한다(4번).
- b. 산출물은 실시간 **알림** 또는 주기 **잔고대사 리포트** 두 형태 모두 지원.

### 4. 제외 tx 등록 시스템

- tx_id 를 입력하면 대사에서 제외되는 화이트리스트. wallet DB 에 별도 테이블
  (예: `reconciliation_exclusions` — tx_id, asset_id, 사유, 등록자, 등록일).
- 등록 주체는 재경팀 → 입력 UI 는 5번 웹에 포함.

### 5. 대사 웹 페이지

- 알림/리포트에서 차이가 보고됐을 때, **원인을 웹에서 바로 추적**할 수 있어야 한다.
- 구성:
  - 자산별 잔고 비교 화면 (DB잔고 vs 온체인잔고 vs 계산잔고)
  - 불일치 tx 목록
  - tx 상세 화면 — 목록에서 tx 를 클릭하면 그 tx 의 온체인 정보(from/to/금액/블록/수수료)와
    OMS 기록(있다면)을 나란히 놓고 대조 ("차이가 났다" → "이 tx 의 이 필드 때문이다"까지
    클릭 몇 번으로 도달하게)
  - 제외 tx 등록 화면(4번)

## 전체 흐름

```
[wallet-tx-scanner]──── 온체인TX ────→ detected_transactions ─┐
                                                              │
[1시간 배치] 온체인잔고 실측(+block_number) ──→ 스냅샷 테이블 ──┤
[OMS] DB잔고 스냅샷 ─────────────────────────────────────────┤
                                                              ▼
                          [대사 엔진]  ① 헬스체크(1-c)  ② 시점 보정(2)  ③ tx 매칭(3)
                                       └ 제외 목록(4) 반영
                                                              ▼
                          불일치 테이블 ──→ 알림 / 리포트 / 웹(5)
```

## 미확정 사항 (설계 진행하며 채움)

- DB잔고(OMS 스냅샷)의 정확한 소스: 어떤 테이블/뷰에서, transaction_status 를 어떻게 구분해서 뽑는지.
- tx 매칭의 OMS 측 대상 테이블(입금/출금 기록)과 매칭 키.
- 불일치 분류(버킷) 상세: MATCHED / ON_CHAIN_ONLY(in·out) / OMS_ONLY / AMOUNT_MISMATCH /
  INTERNAL(우리 지갑 간 이동 — 포워딩·콜드 이동은 OMS 에 없어도 정상) / EXCLUDED(4번).
- 판정 유예(grace window): 감지 직후 OMS 처리 지연으로 인한 가짜 불일치 방지(예: 30분 미매칭 시 확정).
- 알림 채널, 리포트 주기/형식, 웹 접근 제어.
